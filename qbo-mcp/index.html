STEP1

Install Node.js

The runtime your MCP server needs to run.
▼

Node.js is the runtime your MCP server needs in order to run. Install it once like any other app. You never need to touch it again after this step.

  1. Go to nodejs.org
  2. Click the LTS version — "Recommended for most users." It auto-detects your OS
  3. Open the installer and click through: Continue → Continue → Agree → Install
  4. Accept all defaults. Don't change any settings
  5. Click Close when it's done

Verify the installation

This is the only time in the entire process you use a command window. Open it, type one thing, look for a number.

  1. Press Cmd + Space, type Terminal, press Enter
  2. In the black window type: node --version
  3. Press Enter
  1. Press the Windows key, type cmd, press Enter
  2. In the black window type: node --version
  3. Press Enter
Success looks like this

You see a version number — something like v22.14.0. Any number means Node.js is working. Close the window. If it says "command not found" or "not recognized," restart your computer and try again. If it still fails, ask Claude Code: "I installed Node.js but node --version says command not found." He'll fix it.

STEP2

Create your Intuit Developer account

Free. Uses your existing Intuit / QBO login.
▼

Your Intuit developer account is where you create the app that gives you the API keys. It's free, and it uses your existing Intuit / QBO login.

  1. Go to developer.intuit.com
  2. Click Sign In — use your existing Intuit / QBO credentials
  3. Accept the developer terms if prompted
  4. You land on the Developer Dashboard

That's it for this step. We'll create the actual app in Step 4. First we set up the sandbox.

STEP3

Set up your sandbox company

A free private test environment. Don't skip the seeding.
▼

Intuit's sandbox is a private, free test environment. It's where you'll connect Claude first so you can break things without consequences. The catch nobody mentions: the sandbox starts empty. You have to create a company before anything is there.

  1. Inside developer.intuit.com, click the Hub icon in the top right
  2. If you've never been here before, you'll see no companies
  3. Click to add a company → choose which QBO product you want → add it
  4. You can add up to 10 companies in your sandbox
💡
ProTip

Curious about Intuit Enterprise Suite? You can spin one up in your sandbox too. This is the cheapest way to get a feel for IES before you ever recommend it to a client.

Once your sandbox company exists, log into it directly at qbo.intuit.com (make sure you're in sandbox mode) and add a few customers, a few invoices, and a few bills. Without data, the read tools will return empty results — which looks like a failure but isn't. Five minutes of seeding the sandbox saves you an hour of wondering why nothing's happening.

STEP4

Create your app in the developer portal

Each app gets two pairs of keys — sandbox and production.
▼

Now you create the app inside your developer account. Each app gets its own pair of keys — one for sandbox (Development) and one for live data (Production). You'll start with the sandbox keys.

  1. Click + Create an app
  2. Select QuickBooks Online and Payments
  3. Name it anything — "QBO MCP Server" works fine
  4. Under Scopes, check com.intuit.quickbooks.accounting
  5. Click Create app
STEP5

Add your redirect URI

localhost means "this computer, right here."
▼

The redirect URI is how QBO sends authorization back to your computer after you log in. It points at your own machine — not a website you have to own. Get it exactly right or QBO will reject the connection.

  1. Open your new app → click Keys & credentials on the left
  2. Make sure you're on the Development tab (not Production)
  3. Find Redirect URIs and add exactly: http://localhost:3000/callback
  4. Save
What this actually means

localhost means "this computer, right here." Your MCP server will start a tiny temporary web server on port 3000 during the login handshake — just long enough to catch the token and shut back down. The redirect URI must match exactly or QBO refuses to play.

STEP7

Get your keys and paste them into .env

Don't store them anywhere on the way.
▼

Now go grab your keys and paste them straight into the .env file Claude Code just created. Don't store these anywhere else along the way.

The keys live where they live

Don't copy your Client ID and Secret into Notepad, a sticky note, an email draft, or a Google Doc. They live in the developer portal until they live in your .env file. Nowhere else. Keep the developer portal tab open or bookmark it in Notion so you can come back to it. That's the discipline.

Grab the keys

  1. Back in your Intuit Developer app, click Keys & credentials
  2. Confirm you're on the Development tab
  3. Toggle Show credentials
  4. Leave the tab open. You're going to copy → paste, not copy → save → paste

Open the .env file

The .env file starts with a dot which hides it by default. Here's how to see it and open it on each platform.

  1. In Finder press Cmd + Shift + . to show hidden files
  2. Open your qbo-mcp-server folder
  3. Cmd+Click .env → Open With → TextEdit
  1. In File Explorer click View → Show → Hidden items
  2. Open your qbo-mcp-server folder
  3. Right-click .env → Open With → Notepad

Paste your values

QBO_CLIENT_ID=paste_your_client_id_here QBO_CLIENT_SECRET=paste_your_client_secret_here QBO_REDIRECT_URI=http://localhost:3000/callback QBO_ENVIRONMENT=sandbox

Replace the placeholder text after the equals signs with your actual values. Save. Close. Done.

Never share this file

Don't email it. Don't upload it to Google Drive. Don't paste it into any chat window. The keys inside give full access to your QBO account.

First-time login

The first time Claude uses the QBO server, a browser window opens for QBO login. Log in with your Intuit credentials and click Allow. QBO sends you back to localhost:3000/callback, the server captures the token, the browser closes. From that point it's automatic for 100 days.

Already have the native QBO connector? Disconnect it.

If you also have the native QBO connector enabled, Claude will see whichever company you connected him to there. My vote: disconnect and remove the native connector altogether. This MCP connection gives you so much more than that one does. Keeping both will almost assuredly cause confusion.

Prompt 2 — Write tools 
I have an existing QBO MCP server in my qbo-mcp-server folder.
It currently has read tools only. Please add:

1. create_customer
   Parameters: display_name (required), email, phone,
   billing_address (all optional)
   POST /v3/company/{realmId}/customer

2. update_customer
   Parameters: customer_id (required), display_name, email,
   phone, billing_address (all optional)
   Fetch current customer for SyncToken first, then update
   POST /v3/company/{realmId}/customer

3. create_item (product or service)
   Parameters: name (required), type (required — Service,
   Inventory, or NonInventory), unit_price, description,
   income_account_name (all optional)
   POST /v3/company/{realmId}/item

4. create_invoice
   Parameters: customer_ref (required), line_items (required
   — array of description + amount), due_date, send_email
   POST /v3/company/{realmId}/invoice

5. create_bill
   Parameters: vendor_name, amount, category,
   transaction_date (all required), memo (optional)
   POST /v3/company/{realmId}/bill

6. create_account
   Parameters: name, account_type (required — e.g. Income,
   Expense, Bank, Credit Card), account_sub_type, description
   POST /v3/company/{realmId}/account

7. send_invoice_email
   Parameters: invoice_id (required), email (optional)
   POST /v3/company/{realmId}/invoice/{id}/send

8. import_transactions_from_csv
   Parameters: file_path (required), transaction_type
   (required — Expense, Bill, or JournalEntry),
   bank_account_name (required), dry_run (optional boolean)
   Reads CSV, categorizes using Chart of Accounts, imports
   via POST /v3/company/{realmId}/batch

Keep existing OAuth and read tools unchanged. Add clear error
handling on all write operations. Tell me how to restart the
server when done.

STEP12

Switch to production / connect to real books

Keep both sandbox and production keys. Demo is gold.
▼

Once Intuit approves your app, you can grab production keys and point Claude at live data. Two things to know first.

  1. Don't connect to a client's books without their permission.
  2. Keep both sets of keys in your .env so you can demo in sandbox and work in production. I highly recommend this — being able to demo your capabilities to a prospective client is huge. Very few accountants can do this right now.

Pull your production keys

  1. Once approved, go to your app → Keys & credentials → Production tab
  2. Copy your production Client ID and Client Secret
  3. Ask Claude Code to update your .env with both sandbox and production keys, and tell him you want to be able to switch between them. Give him the keys when he asks. He'll handle the file change.
  4. Restart Claude Desktop
Accessing client books — two paths

Option A. If you're already the accountant user in the client's QBO: log in with your own Intuit credentials when the OAuth screen appears. Intuit shows you a company picker with every company you have access to. Pick the client.

Option B. If you're not their accountant user: ask Claude Code to generate an authorization link. Send it to the client. They click it, log in with their own credentials, and authorize. Tokens come back to your server. They never share their password.
